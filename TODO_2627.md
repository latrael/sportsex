## Phase 1 — Ingestion · 3–4 days

- [x] **R1.1** Typed FPL client: `bootstrap-static`, `fixtures`, `event/{gw}/live`. Retry with backoff, a real User-Agent, and raw response retention so any parse can be replayed without refetching.
      Lives in `apps/web/src/lib/fpl/`: `types.ts` (zod schemas), `parse.ts` (pure parsers + `parseByEndpoint` replay), `client.ts` (`FplClient`).
      Design decisions worth knowing:
      - **Fetching and parsing are separate.** Every parser takes plain `unknown`, so a payload archived today can be re-parsed by tomorrow's code with no network call. The client returns `{ data, raw }` and takes an `onRaw` hook — point it at a table or blob store in R1.5 and the whole backfill becomes replayable.
      - **Every schema is `.passthrough()`.** FPL adds fields mid-season without notice (`defensive_contribution` is a 25/26 arrival). Unknown fields must never fail a parse.
      - **Decimal fields stay strings** (`"expected_goals": "8.23"`), exactly as the wire carries them. Parsing them to JS numbers here would put a float in front of every Decimal column; Prisma takes the string directly. This is R2.4's rule applied one layer earlier.
      - Retry covers 408/425/429/5xx, transport errors, and truncated bodies (a half-received 2 MB payload is a transport problem, not a schema problem). Full-jitter exponential backoff, `Retry-After` honoured up to 60s and a hard fail beyond that. 4xx other than those listed fail immediately. `random`, `sleep` and `fetch` are all injectable, so backoff is asserted exactly and no test spends real seconds.
      **Accept:** met. 39 unit tests, no network, against four checked-in payload fixtures in `src/test/fixtures/fpl/` — genuine trimmed 25/26 responses, referentially closed (every `element.team` resolves to a team in the file, every `explain[].fixture` to a fixture in `fixtures.sample.json`). GW33 is a real double gameweek, so the per-fixture attribution R1.4 depends on is already under test. Full suite 119 passed (80 → 119); typecheck and lint clean.
      *Verified separately against the live API* (not part of the suite): 20 teams, 841 players, 38 events, 380 fixtures, and GW1/26/33/38 live all parse — 79 players with 2+ fixtures in GW26, 248 in GW33.

- [x] **R1.2** `bootstrap-static` → clubs, players, gameweeks. Identity keyed on `fpl_id` + `opta_code`; store `birth_date`, `squad_number`, `team_join_date`, `status`, `chance_of_playing_next_round`.
      *Data reality found in R1.1, applies to the schema:* `squad_number` is null for **all 841** players in the 25/26 dataset — the field exists but FPL never populates it, so don't make it non-null or plan to use it for identity. `birth_date` is null for 19 players and `chance_of_playing_next_round` for 211. `opta_code` is populated for all 841 (`p118748` form), which is what makes it usable as the cross-season identity key.
      **Correction to that last point, found by running it:** `opta_code` is exactly `"p" + code` for **all 841** players. It is derived from `code`, not independent of it, so it corroborates nothing about identity that `code` doesn't already say — the identity key is `code` alone. Stored and kept unique regardless, because it's the join key into other providers' Opta data, which is what R1.6's independent cross-check will need.
      Schema (`prisma/migrations/20260723062333_r1_2_reference_tables`): `season`, `club`, `club_season`, `fpl_player`, `player_season`, `gameweek`. Code in `apps/web/src/lib/ingest/`.
      Design decisions worth knowing:
      - **Identity and season membership are separate tables.** FPL renumbers every `element.id` and `team.id` in early August, so those ids are useless as keys — but they're also the *only* keys `fixtures` and `event/{gw}/live` carry. Splitting `fpl_player`/`club` (stable `code`) from `player_season`/`club_season` (season-scoped `fplId`) gives both: cross-season identity, and a `(seasonId, fplId)` mapping that still resolves a 25/26 payload after the rollover has reassigned those ids. Without it, replaying the backfill post-August (R1.7) would silently attribute stats to the wrong players. This is the main reason the table count is 6 and not 3.
      - **Planning is pure and separate from writing**, same split as R1.1's parse/fetch. `planBootstrap(payload)` → rows, no clock, no DB, no network. All 22 planner tests run against the checked-in fixture.
      - **Hard failures vs warnings is an operational choice, not a taste one.** Anything that would corrupt data or break a unique constraint (element pointing at an absent team, duplicate `code`/`opta_code`, a date in a format FPL has never used) throws and stops the ingest. Per-row noise a transfer window can produce (`team_code` disagreeing with `team`) comes back as a warning, because failing there would take the daily cron down for one inconsistent row.
      - **Writes are diffed, not upserted.** Prisma's `upsert` always issues an UPDATE, so a re-run would report ~900 writes with nothing changed and R1.7 could never be honestly satisfied. `applyDiff` (`src/lib/ingest/diff.ts`) reads first and writes only what moved, normalising `Date` and `Decimal` so Prisma's return types don't read as changes. Ingestion never deletes: FPL drops elements mid-season and their history stays valid.
      - `Position` is mapped from `element_types[].singular_name_short`, not from the numeric `element_type` id, which is FPL's to renumber.
      **Accept:** met. Live run against the real API: **20 clubs, 841 players, 38 gameweeks**, 0 warnings, GKP 97 / DEF 270 / MID 379 / FWD 95, squads 31 (FUL) to 52 (SUN). Zero name-based matching in the path. A second sync of the same payload reported **0 created, 0 updated, 900 unchanged in 18ms** — R1.7 is already true for this endpoint. 33 new tests (22 planner, 11 sync), suite 119 → 152; typecheck and lint clean; `prisma migrate deploy` applies both migrations to an empty database with no drift.

## Phase 3 — Valuation engine · 4–5 days

- [ ] **R3.1** Per-fixture performance score. Attack (goals, assists, xG, xA), defence (clean sheets, saves, tackles, interceptions, clearances, recoveries, xGC), discipline penalties, bonus. **z-scored within position group** so GK/DEF/MID/FWD land on comparable distributions.
      **Accept:** pure function, unit-tested, mean ≈ 0 and sd ≈ 1 within each position group across 25/26.

- [ ] **R3.2** Form: EWMA over recent fixtures, half-life ≈ 4 matches.
      **Accept:** unit test on a hand-computed sequence.

- [ ] **R3.3** Baseline with shrinkage: `(n₉₀·rate + k·cohort_prior) / (n₉₀ + k)`, k ≈ 6.
      **Accept:** a player with 90 minutes sits within 10% of the positional median; convergence to own rate is monotone in minutes.

- [ ] **R3.4** Availability multiplier from expected-minutes share (EWMA over last 6 fixtures, blended with starts rate) and FPL `status` / `chance_of_playing_next_round`.
      **Accept:** a player who stops appearing decays toward the floor over ~6 gameweeks; an `i`/`s` status player drops immediately.

- [ ] **R3.5** Compose `fairValue()`: `clamp(P_pos_base · exp(λ·(0.6·B + 0.4·F)) · M, 10, 1500)`.
      **Accept:** invariant 6 — pure, deterministic, no clock, no randomness.

- [ ] **R3.6** Backtest harness: replay 25/26 gameweek by gameweek, dump the price distribution at GW1/10/20/38.
      **Accept:** a reproducible report artifact, regenerable from one command.

- [ ] **R3.7** Calibration pass on λ, position weights, and half-life. This is the real work of the phase, not polish.
      **Accept:** all four position groups have players in the top 100 by price; the 5th–95th percentile price ratio is ≥10×; a manual sanity list (elite striker near the top, first-choice keeper above a fringe forward, injured player decaying) reads correctly.

- [ ] **R3.8** **Owner decision:** position weighting — what a clean sheet is worth relative to a goal. Determines whether this is a fantasy-football clone or its own thing. Decide deliberately; don't default to FPL scoring.
      **Accept:** decision recorded here with rationale, weights committed.

## Phase 4 — AMM trading engine · 4–5 days

- [ ] **R4.1** Bonding curve: `P(q) = V·exp((q−q₀)/L)`, cost = the integral, fee ~0.5% per side.
      **Accept:** invariant 1 — buy `n` then sell `n` loses exactly the fee, for all `n`, all `V`, all starting `q`.

- [ ] **R4.2** Double-entry ledger. Balance is the sum of a user's entries, never a stored mutable field.
      **Accept:** balances derivable by query; a reconciliation query returns zero discrepancies.

- [ ] **R4.3** Rewrite `POST /api/orders` on the curve: execution and price formation in one transaction. Removes the `preTradeDemand` ordering bug by construction.
      **Accept:** the §2 exploit sequence (buy 1000 → buy 1 → sell 1001) nets a loss equal to fees. Written as a regression test.

- [ ] **R4.4** Durable rate limiter replacing the in-process `Map` in `src/lib/cooldown.ts` (DB or Upstash).
      **Accept:** the limit holds across separate serverless invocations.

- [ ] **R4.5** Fuzz + concurrency suite: 10k random trades across 100 users, with parallel writers.
      **Accept:** invariants 2 and 5 — coins conserved to the last unit, no negative balance, no lost update.

- [ ] **R4.6** Seed initial inventory `q₀` and per-player liquidity depth `L`, scaled by float.
      **Accept:** price impact of a 1% -of-float order is within a documented target band across the whole market.
